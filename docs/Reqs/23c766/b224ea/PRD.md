# PRD — E2E: Test data backup and restore workflow

> 所属需求：数据备份与恢复

## 用户故事
As a 项目管理员，I want to 通过 E2E 测试验证数据备份与恢复的完整流程，So that 确保用户数据导出、清空、导入功能在真实浏览器环境下稳定可靠，防止数据丢失风险。

## 功能需求
- 测试用例 1：正常流程测试
  - 访问页面多次生成测试数据（至少 5 条访问记录）
  - 点击导出按钮，验证浏览器下载 JSON 文件
  - 解析下载的 JSON 文件，验证格式正确且包含所有记录
  - 点击清空按钮，确认二次确认对话框出现
  - 确认清空操作，验证数据被清空且计数归零
  - 上传之前导出的 JSON 文件，验证数据恢复成功
  - 验证恢复后的访问次数与导出前一致

- 测试用例 2：异常场景测试
  - 上传非法 JSON 文件（格式错误/字段缺失），验证错误提示出现
  - 上传非 JSON 文件（如 .txt），验证文件类型校验生效
  - 点击清空按钮后取消确认，验证数据未被清空

- 测试用例 3：边界情况测试
  - 在无数据状态下点击导出，验证下载空数组 JSON 文件
  - 重复导入同一份数据文件，验证数据不重复累加（覆盖或去重逻辑）
  - 导入包含大量数据的 JSON 文件（>100 条记录），验证导入成功且性能可接受

- 测试截图记录
  - 每个关键步骤自动截图保存到 `tests/e2e/screenshots/` 目录
  - 截图命名格式：`backup-restore-{step}-{timestamp}.png`

## 验收标准
- [ ] 测试用例 1 执行时间 ≤ 30s，所有断言通过
- [ ] 导出的 JSON 文件包含字段：`visits`（数组）、`total_count`（数字）、`export_time`（ISO 8601 时间戳）
- [ ] 清空操作后，页面显示访问次数为 0，再次导出的 JSON 文件 `visits` 数组长度为 0
- [ ] 导入操作后 500ms 内页面数据刷新，访问次数与导入文件中的 `total_count` 一致
- [ ] 上传非法 JSON 时，300ms 内显示错误提示 Toast，文案包含「格式错误」或「解析失败」
- [ ] 取消清空确认后，数据保持不变，访问次数与操作前一致
- [ ] 空数据导出的 JSON 文件大小 ≤ 1KB，格式符合 schema
- [ ] 重复导入相同数据时，访问次数不重复累加（最终值 = 导入文件中的 `total_count`）
- [ ] 导入 100 条记录的文件时，导入完成时间 ≤ 2s，页面无卡顿
- [ ] 每个测试步骤生成截图文件，文件名包含步骤标识和时间戳，图片分辨率 ≥ 1280x720
- [ ] 测试失败时，自动保存失败截图和浏览器控制台日志到 `tests/e2e/failures/` 目录
- [ ] 所有测试用例在 Chromium、Firefox、WebKit 三个浏览器内核下均通过

## 边界条件（不做的事）
- 不包含：性能压测（如导入 10000+ 条记录的极限测试）
- 不包含：多用户并发导入导出的竞态条件测试
- 不包含：数据加密/解密功能测试（当前版本数据明文存储）
- 不包含：跨浏览器 Cookie/LocalStorage 数据迁移测试
- 暂不支持：移动端触摸手势的 E2E 测试（仅测试桌面端点击操作）
- 超出范围：后端 API 单元测试（由独立的 pytest 测试覆盖）
- 超出范围：数据库备份恢复的底层逻辑测试（E2E 仅验证前端交互）
